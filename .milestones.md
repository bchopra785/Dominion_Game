## General Advice

Plan your sprints wisely. Leave plenty of time for merging, code reviews, testing, and bug fixing. If you plan to fully complete all 4 stories right before the end of the 3rd sprint, you will have no time to test or get feedback, and will likely need to use a 4th sprint to finish the milestone, which in turn will crunch the schedule for later milestones. You will also likely receive lower individual sprint scores due to lack of work being merged to main each sprint.

Think about how to parallelize work. For example, if you plan to fully complete one major story per week, without working on the others at all, you will not have enough work to keep everyone busy each week, and your last feature will end up squeezed for time when your first features take longer than expected.

Do not over-design up front. When you first start to code on a larger project, it is easy to get paralyzed by the need to fully design the system before starting. Do not let this paralyzation happen - some initial design is useful, but when you start to go around in circles, the best thing to do is to start coding to learn more about the problem.

Do not under-design up front. Conversely, if your team does not discuss an initial approach at all, multiple people will end up duplicating work or working in incompatible ways. Talk about, and try to agree on some high level components, without slipping into over-designing.

Communicate often. Your initial plans, ideas, and designs will change as you get into the project. Communicate roadblocks, changes, or new ideas to your team as soon as possible so that the team can work in a coordinated fashion.

Test and refactor early and often. Practice Test Driven Development, or, failing this, write a small amount of code and test it before moving on. When your code is not easily testable, refactor it to modularize. Do not be afraid to introduce helper classes or functions. The more you encapsulate common or complicated functionality in well tested classes or functions, the easier the later projects in this class will be.

Work in small increments. You are used to completing entire projects at once, submitting them, and moving on to the next project. This is a bad habit that you should break - identify small changes or additions, complete them (and test them), then commit and merge those changes to main, or at least to a branch. Working on huge changes for a long time, and then merging them all at once is disruptive to the rest of your team, and will cause hard-to-find bugs that slow you all down.

Use code generation tools productively. The amount of work in this milestone is designed with the expectation that you will use code generation tools to assist with the implementation and testing. However, the body of work here is likely too large and complex to wholesale delegate to a coding agent. Additionally, part of the required challenge of the assignment is figuring out how to split up and parallelize work across multiple team members. Experiment with code generation tools to find the most effective and efficient ways to leverage them, but do not expect to be successful if you attempt to completely outsource the project to them.

## Story 3: Big Money Player

In this story, you will implement a Player that follows a well-defined strategy. Choose from one of the strategies below (basic, advanced 1, advanced 2), or a combination of the three.

### "Big Money" Player Basic Strategy

A "big money" player will always buy a Framework card if possible; otherwise, it will buy the highest valued money card that it can afford.

### (Optional) Advanced Strategy 1: Boost Hand Expected Value

A "big money" player can follow a more sophisticated strategy: Frameworks are only purchased if the current hand can afford it, AND if the expected money value of a hand from the player's deck is close to or greater than the cost of a Framework.

The end result of following this strategy is that more money will be purchased initially, which will allow the player's deck to maintain purchasing power even after it is diluted by Framework purchases.

### (Optional) Advanced Strategy 2: Do not end the game if losing

A "big money" player can keep track of the number of victory points purchased by itself and its opponent(s), and avoid purchasing the last Framework (and instead purchase a Module or Method) if they are not winning.

### Story 3 deliverables

1. An implementation of Player which follows at least the "basic" strategy above
	 1. The advanced strategies are completely optional.
2. Unit test coverage of the big money player class

## Code review

During each major milestone, your group will need to schedule a code review with either the instructor or your team's assigned TA. The output of the code review is a combination of required and optional changes. Your team must implement the required changes in order to fully complete the milestone.

### Timing of the code review

The best time for a code review is during the second sprint of a milestone, or early in the third sprint. This allows time for some changes from the milestone to be landed on the main branch, but also leaves enough time afterward to implement the required changes.

### Preparation for the code review

Create a branch named code-review-milestone-<#> (replacing <#> with the number of the milestone) from the git tag that marked the release from your previous milestone.

Open a pull request from the tip of your main branch into this new branch. The description of the pull request should include a high level summary of the changes to be reviewed, and any other information that will help the reviewer navigate the changes.

Assign the course staff member who will be reviewing your code and the instructor as the "Reviewers" on the pull request. The code reviewer will leave their feedback on this pull request.

### Code review process

Code reviews are performed synchronously - the reviewer and the team should be together (in person if at all possible, virtually if not) for the review. The entire team is strongly encouraged to attend; however, if schedule conflicts preclude this, a time should be chosen when at least 2 team members can attend.

The reviewer will "drive" (navigate the code on their laptop). The code review is intended to be collaborative and interactive - discussion between the reviewer and team is encouraged. The code reviewer will leave comments on the pull request during the review to summarize discussions and document suggestions.

Required changes will be explicitly labeled as such - any comment on a code review that is not clearly specified as required is optional. The code reviewer may take a short amount of time after the review to mark the set of required changes, and will notify the team when the feedback is ready to be acted upon.

## Story 2: Network Player

In this story, you will create the client and server for a network-based player.

The "player client" will implement the Player interface from the ATG API, and be used directly by the Engine. It will not contain any decision-making logic, and instead will request decisions from a remote HTTP server - the "player server". The player server will contain all the necessary decision-making logic, and respond with the chosen decision. The player client itself should not contain any player logic, and should be able to work with any player server which follows the server API specifications (found below).

The "player server" will be an HTTP server. It will use an "RPC" (remote procedure call) API style. It will accept POST requests with JSON bodies that represent the game state and possible decisions, and respond with a chosen decision.

### Specifications

#### HTTP Server

The player server should expose two HTTP endpoints:

- /decide (POST)
	- Expects a JSON body formatted according to the "DecisionRequest" schema below.
	- Successful (200) responses should include a JSON body formatted according to the "DecisionResponse" schema below.
	- Unsuccessful responses should use appropriate 4xx or 5xx HTTP response codes.
- /log-event (POST)
	- Expects a JSON body formatted according to the "LogEventRequest" schema below.
	- Successful (200) responses should have empty bodies.
	- Unsuccessful responses should use appropriate 4xx or 5xx HTTP response codes.

#### Request / Response formats

The formats below are specified using the JSON Schema format. (Be sure to read up on the basics of JSON Schema so you can understand the expected structure!)

DecisionRequest

```json
{
	"title": "DecisionRequest",
	"description": "An object representing a request for a player to choose a decision",
	"type": "object",
	"properties": {
		"state": {
			"description": "A serialized GameState object",
			"type": "GameState"
		},
		"options": {
			"description": "The list of possible Decisions",
			"type": "array",
			"items": {
				"description": "A serialized Decision object",
				"type": "Decision"
			}
		},
		"reason": {
			"description": "A serialized Event object",
			"type": "Event"
		},
		"player_uuid": {
			"description": "A unique identifier for the player making the request",
			"type": "string"
		}
	}
}
```

DecisionResponse

```json
{
	"title": "DecisionResponse",
	"description": "An object representing a decision chosen by the player",
	"type": "object",
	"properties": {
		"decision": {
			"description": "A serialized Decision object",
			"type": "Decision"
		}
	}
}
```

LogEventRequest

```json
{
	"title": "LogEventRequest",
	"description": "An object representing a game event that can be logged",
	"type": "object",
	"properties": {
		"state": {
			"description": "A serialized GameState object",
			"type": "GameState"
		},
		"event": {
			"description": "A serialized Event object",
			"type": "Event"
		},
		"player_uuid": {
			"description": "A unique identifier for the player this event is being sent to",
			"type": "string"
		}
	}
}
```

In the schema above, several objects have "types" that correspond with types provided by the ATG API. These types are already configured to serialize themselves to JSON in the appropriate schemas, and so the details of those schemas are not included here.

### Resources & Recommendations

#### Server language

Your HTTP server can be written in any language - this is one of the benefits of modularization at the system level. However, it is highly recommended to implement it in Java, so you can use your existing player implementations to make decisions, rather than needing to implement a new player from scratch. In Java, you also benefit from using all of the classes provided by the ATG API. But, if you would prefer to try another language, you are free to do so.

#### Java Server Framework

There are many ways to build an HTTP server in Java. The recommended approach for this project is to use Spring. Spring is a very heavyweight, opinionated Java framework that makes it possible to implement a good, compliant webserver with very little code. The price of this, though, is that the Spring framework itself is large and complex. You only need to learn a tiny slice of it in order to get a webserver working, but identifying which tiny slice is a challenge.

#### Reference Repository

Here is a sample repo which contains a multi-module Maven project, with a Spring sub-module, and an example of installing the Guava module needed to support parsing/serializing Guava objects to JSON: https://github.com/brandeis-cosi-103a/spring-server-example. This is a very close match to what you will need to do to get a Spring server running!

## Code review

During each major milestone, your group will need to schedule a code review with either the instructor or your team's assigned TA. The output of the code review is a combination of required and optional changes. Your team must implement the required changes in order to fully complete the milestone.

### Timing of the code review

The best time for a code review is during the second sprint of a milestone, or early in the third sprint. This allows time for some changes from the milestone to be landed on the main branch, but also leaves enough time afterward to implement the required changes.

### Preparation for the code review

Create a branch named code-review-milestone-<#> (replacing <#> with the number of the milestone) from the git tag that marked the release from your previous milestone.

Open a pull request from the tip of your main branch into this new branch. The description of the pull request should include a high level summary of the changes to be reviewed, and any other information that will help the reviewer navigate the changes.

Assign the course staff member who will be reviewing your code and the instructor as the "Reviewers" on the pull request. The code reviewer will leave their feedback on this pull request.

### Code review process

Code reviews are performed synchronously - the reviewer and the team should be together (in person if at all possible, virtually if not) for the review. The entire team is strongly encouraged to attend; however, if schedule conflicts preclude this, a time should be chosen when at least 2 team members can attend.

The reviewer will "drive" (navigate the code on their laptop). The code review is intended to be collaborative and interactive - discussion between the reviewer and team is encouraged. The code reviewer will leave comments on the pull request during the review to summarize discussions and document suggestions.

Required changes will be explicitly labeled as such - any comment on a code review that is not clearly specified as required is optional. The code reviewer may take a short amount of time after the review to mark the set of required changes, and will notify the team when the feedback is ready to be acted upon.
